## 1. Что такое поток выполнения?

**Ответ:**

Поток выполнения (thread) — это независимая последовательность команд, выполняющихся в рамках одного процесса. Потоки позволяют программе выполнять несколько задач параллельно.

**Основные характеристики:**
- Потоки разделяют общую память процесса
- Каждый поток имеет свой стек вызовов
- Потоки могут выполняться одновременно на многоядерных процессорах
- Используются для параллельного выполнения задач

**Пример из лабораторной работы:**
В нашем приложении операции сохранения и загрузки файлов выполняются в отдельных потоках, что позволяет основному потоку продолжать работу, не дожидаясь завершения операций ввода-вывода.

## 2. Чем отличается Thread от Runnable?

**Ответ:**

`Thread` и `Runnable` — это два способа создания потоков в Java, которые имеют принципиальные различия:

### Thread (класс)
- Это класс, который можно расширить (extends Thread)
- Содержит метод `run()`, который нужно переопределить
- Объект класса-наследника сам является потоком
- Недостаток: в Java нет множественного наследования, поэтому нельзя расширить другой класс

**Пример:**
```java
class MyThread extends Thread {
    public void run() {
        System.out.println("Поток запущен");
    }
}
// Использование
MyThread thread = new MyThread();
thread.start();
```

### Runnable (интерфейс)
- Это интерфейс с одним методом `run()`
- Можно реализовать его в любом классе (implements Runnable)
- Объект передаётся в конструктор Thread
- Преимущество: класс может наследоваться от другого класса

**Пример:**
```java
class MyTask implements Runnable {
    public void run() {
        System.out.println("Задача выполняется");
    }
}
// Использование
Thread thread = new Thread(new MyTask());
thread.start();
```

**Вывод:** Использование Runnable более гибкое и предпочтительнее в большинстве случаев.

## 3. Почему нельзя вызывать метод run() напрямую?

**Ответ:**

Прямой вызов метода `run()` не создаёт нового потока — код просто выполнится в текущем потоке.

### Что происходит при вызове run():
```java
FileSaveTask task = new FileSaveTask(employees, "file.txt");
task.run(); // ❌ Выполняется в главном потоке
```
- Метод `run()` выполняется как обычный метод
- Код блокирует выполнение программы
- Параллелизм не достигается

### Что происходит при вызове start():
```java
Thread thread = new Thread(task);
thread.start(); // ✅ Создаётся новый поток
```
- Метод `start()` создаёт новый поток выполнения
- JVM автоматически вызывает `run()` в новом потоке
- Главный поток продолжает работу параллельно

**Важно:** Метод `start()` можно вызвать только один раз для каждого объекта Thread. Повторный вызов приведёт к исключению `IllegalThreadStateException`.

## 4. Зачем нужны коллекции?

**Ответ:**

Коллекции — это структуры данных для хранения и управления группами объектов.

### Основные причины использования коллекций:

1. **Динамический размер**
   - Автоматически расширяются при добавлении элементов
   - Не нужно заранее знать количество элементов

2. **Готовые методы**
   - Добавление, удаление, поиск элементов
   - Сортировка, фильтрация
   - Итерация по элементам

3. **Типобезопасность**
   - Generics (обобщения) обеспечивают проверку типов на этапе компиляции
   - Пример: `ArrayList<Employee>` — можно хранить только объекты Employee

4. **Различные реализации**
   - ArrayList — для быстрого доступа по индексу
   - LinkedList — для частых вставок/удалений
   - HashSet — для хранения уникальных элементов
   - HashMap — для хранения пар ключ-значение

**Пример из лабораторной:**
```java
ArrayList<Employee> employees = new ArrayList<>();
employees.add(new Employee("Иванов", 30, "Программист", 80000));
employees.add(new Employee("Петрова", 28, "Менеджер", 65000));
// Размер увеличивается автоматически
```

## 5. Чем ArrayList отличается от массива?

**Ответ:**

### Массив (Array):
```java
Employee[] employees = new Employee[10]; // Фиксированный размер
```

**Характеристики:**
- ❌ Фиксированный размер (нельзя изменить после создания)
- ✅ Быстрый доступ по индексу O(1)
- ❌ Нет встроенных методов для добавления/удаления
- ✅ Может хранить примитивные типы (int, double)
- ❌ Нужно вручную управлять размером

### ArrayList:
```java
ArrayList<Employee> employees = new ArrayList<>(); // Динамический размер
```

**Характеристики:**
- ✅ Динамический размер (автоматически расширяется)
- ✅ Быстрый доступ по индексу O(1)
- ✅ Множество встроенных методов (add, remove, contains, sort)
- ❌ Хранит только объекты (для примитивов используются обёртки)
- ✅ Автоматическое управление памятью

### Сравнительная таблица:

| Характеристика | Массив | ArrayList |
|---------------|--------|-----------|
| Размер | Фиксированный | Динамический |
| Типы данных | Примитивы и объекты | Только объекты |
| Методы | Нет встроенных | add(), remove(), size(), и др. |
| Производительность | Немного быстрее | Небольшой overhead |
| Удобство | Низкое | Высокое |

**Вывод:** ArrayList удобнее для большинства задач, массивы используют когда критична производительность и размер известен заранее.

## 6. Что такое try-with-resources?

**Ответ:**

`try-with-resources` — это конструкция Java, которая автоматически закрывает ресурсы после использования.

### Синтаксис:
```java
try (ResourceType resource = new ResourceType()) {
    // Работа с ресурсом
} catch (Exception e) {
    // Обработка исключений
}
// Ресурс автоматически закрыт
```

### Пример из лабораторной работы:
```java
try (BufferedWriter writer = new BufferedWriter(new FileWriter(filename))) {
    for (Employee emp : employees) {
        writer.write(emp.toFileString());
        writer.newLine();
    }
} catch (IOException e) {
    System.err.println("Ошибка: " + e.getMessage());
}
// writer.close() вызывается автоматически
```

### Преимущества:

1. **Автоматическое закрытие**
   - Не нужно вручную вызывать `close()`
   - Ресурс закрывается даже при исключении

2. **Безопасность**
   - Невозможно забыть закрыть ресурс
   - Предотвращение утечек памяти

3. **Чистый код**
   - Меньше кода
   - Нет блока `finally`

### Старый способ (до Java 7):
```java
BufferedWriter writer = null;
try {
    writer = new BufferedWriter(new FileWriter(filename));
    // Работа с файлом
} catch (IOException e) {
    e.printStackTrace();
} finally {
    if (writer != null) {
        try {
            writer.close();
        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}
```

**Требование:** Ресурс должен реализовывать интерфейс `AutoCloseable`.

## 7. Какие ошибки могут возникать при работе с файлами?

**Ответ:**

### Основные типы ошибок:

#### 1. IOException (общие ошибки ввода-вывода)
**Причины:**
- Файл не найден (FileNotFoundException)
- Нет прав доступа
- Диск заполнен
- Неожиданное прерывание операции

**Пример обработки:**
```java
try (BufferedReader reader = new BufferedReader(new FileReader("file.txt"))) {
    // Чтение файла
} catch (FileNotFoundException e) {
    System.err.println("Файл не найден: " + e.getMessage());
} catch (IOException e) {
    System.err.println("Ошибка чтения: " + e.getMessage());
}
```

#### 2. FileNotFoundException
**Причины:**
- Файл не существует
- Неправильный путь
- Опечатка в имени файла

**Решение:**
- Проверять существование файла перед чтением
- Создавать файл при сохранении

#### 3. SecurityException
**Причины:**
- Нет прав на чтение/запись
- Политика безопасности запрещает доступ

#### 4. NumberFormatException (при чтении данных)
**Причины:**
- Некорректный формат данных в файле
- Попытка преобразовать текст в число

**Пример из лабораторной:**
```java
try {
    int age = Integer.parseInt(parts[1]);
} catch (NumberFormatException e) {
    System.err.println("Некорректный возраст в файле");
}
```

#### 5. OutOfMemoryError
**Причины:**
- Попытка загрузить слишком большой файл
- Недостаточно памяти

### Рекомендации по обработке:

1. Всегда использовать `try-catch` для операций с файлами
2. Использовать `try-with-resources` для автоматического закрытия
3. Предоставлять понятные сообщения об ошибках
4. Валидировать данные перед записью
5. Проверять существование файла перед чтением

## 8. Почему операции ввода-вывода желательно выносить в отдельный поток?

**Ответ:**

Операции ввода-вывода являются медленными (I/O-bound), и их вынос в отдельный поток даёт несколько важных преимуществ:

### 1. Неблокирующий интерфейс
**Проблема:** Операции с файлами могут занимать значительное время (особенно при работе с большими файлами или медленными накопителями).

**Решение:** Главный поток продолжает работу, пока данные сохраняются/загружаются.

```java
// ❌ Без потоков - программа зависает
saveToFile(); // Блокирует выполнение на время сохранения
showMenu(); // Ждёт завершения сохранения

// ✅ С потоками - программа работает
Thread saveThread = new Thread(new FileSaveTask(data, file));
saveThread.start();
showMenu(); // Выполняется сразу
```

### 2. Отзывчивость приложения
- Пользователь может продолжать работать с программой
- Нет «зависания» интерфейса
- Можно показывать индикатор прогресса

### 3. Эффективное использование ресурсов
- Процессор не простаивает во время ожидания I/O операций
- Можно выполнять другие задачи параллельно

### 4. Изоляция ошибок
- Ошибки в потоке I/O не влияют на главный поток
- Можно обработать исключения отдельно

### 5. Масштабируемость
- Можно запускать несколько операций одновременно
- Лучше использует многоядерные процессоры

### Пример из лабораторной работы:

```java
private static void saveToFile() {
    // Создаём задачу
    FileSaveTask saveTask = new FileSaveTask(manager.getEmployees(), FILENAME);
    
    // Запускаем в отдельном потоке
    Thread saveThread = new Thread(saveTask);
    saveThread.start();
    
    // Главный поток может продолжать работу
    // Или дождаться завершения через join()
    try {
        saveThread.join(); // Опционально
    } catch (InterruptedException e) {
        System.err.println("Поток прерван");
    }
}
```

### Когда НЕ нужен отдельный поток:
- Файлы очень маленькие (несколько строк)
- Операция критична для продолжения работы
- Простые консольные приложения без требований к отзывчивости

### Вывод:
Использование потоков для I/O операций — это хорошая практика, которая делает приложение более отзывчивым и профессиональным, особенно при работе с большими объёмами данных.
